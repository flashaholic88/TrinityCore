## 2026-07-16 - Reusable Map Update Scratchpads
**Learning:** In highly concurrent, multi-threaded C++ projects like TrinityCore, map instances are often updated sequentially by a dedicated MapUpdater thread pool where each individual `Map` is processed by one thread at a time. This thread-safety model allows us to declare private member variables on the `Map` class to act as scratchpads for updating and unit visitation loops. Doing so avoids repetitive heap allocation and deallocation of temporary `std::vector` and `std::unordered_set` structures inside the `Map::Update` hot path (which executes on every single map tick).
**Action:** Always identify hot path loops in game map or entity tick functions that perform local collection allocations, and convert them to reusable class member scratchpads. Ensure they are cleared at the start of use, and cleared at the end of the method to avoid retaining dangling pointers to objects that might be deleted.
