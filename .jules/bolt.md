## 2026-07-16 - Heap Churn Elimination in Map::Update Hot Path
**Learning:** In TrinityCore, `Map::Update` runs on every update tick for every map instance with active players or units. Because `Map` instances are ticked sequentially (one thread at a time per map), allocating scratchpad containers locally (`std::vector` and `std::unordered_set`) results in constant heap allocation and deallocation overhead. Converting these local vectors/sets to class member variables and reusing them via `.clear()` preserves allocated capacity, avoiding heap churn entirely.
**Action:** Always identify extremely high-frequency hot path loops that instantiate dynamic containers, and refactor them to use pre-allocated, reusable member variables when thread safety is guaranteed by sequential execution.
